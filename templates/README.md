# __PROJECT_NAME__

Serverless infrastructure on [Scaleway](https://www.scaleway.com), managed with
Terraform and deployed by GitHub Actions. Generated by
[create-serverless-app](https://github.com/Gambi97/create-serverless-app).

## What lives here

This repository contains **infrastructure only** (no application code):

| Resource | Purpose |
|---|---|
| Registry namespace | Private container registry for your app images |
| Container namespace + Serverless Container | Runs your app (scale-to-zero) |
| Serverless SQL Database (Postgres) | One database per environment |

Two environments, `staging` and `prod`, isolated via Terraform workspaces.
State is stored remotely in a Scaleway Object Storage bucket. Application
secrets live in [Infisical](https://infisical.com), never in this repo.

## How deploys work

- **Pull request to `main`** → `terraform-plan.yml` runs `fmt`, `validate` and
  `plan` for both environments. Review the plan in the PR checks.
- **Push / merge to `main`** → `terraform-apply.yml` applies **staging** first,
  then waits for manual approval (GitHub Environment `production`) and applies
  **prod**. After each apply the database endpoint is synced to Infisical as
  `DATABASE_URL`.

## First deploy checklist

1. **Push an application image.** Build your app's Docker image and push it to
   the registry endpoint (see `terraform output registry_endpoint` or the
   Scaleway console after the first apply):

   ```sh
   docker build -t rg.__REGION__.scw.cloud/__PROJECT_NAME__-staging/app:latest .
   docker push rg.__REGION__.scw.cloud/__PROJECT_NAME__-staging/app:latest
   ```

2. **Point the container at the image.** Set `container_image` in
   `staging.tfvars` / `prod.tfvars` and open a PR. Until then, only the
   registry and the database are provisioned (the container is gated on an
   image being available).

3. **Fill in the real secrets in Infisical.** The bootstrap seeded
   placeholders: `BASIC_AUTH_USER` / `BASIC_AUTH_PASSWORD` (staging) and
   `DATABASE_URL` (both, auto-updated after each apply). Your app must read
   these from its environment.

4. **Enforce Basic Auth in the app.** On staging the container receives
   `BASIC_AUTH_ENABLED=true`; the app is expected to protect itself with the
   Infisical-provided credentials.

## Running Terraform locally (optional)

CI is the source of truth, but you can plan locally:

```sh
cp backend.hcl.example backend.hcl
export AWS_ACCESS_KEY_ID=<scaleway-access-key>
export AWS_SECRET_ACCESS_KEY=<scaleway-secret-key>
export SCW_ACCESS_KEY=<scaleway-access-key>
export SCW_SECRET_KEY=<scaleway-secret-key>
export SCW_DEFAULT_PROJECT_ID=<project-id>
export SCW_DEFAULT_ORGANIZATION_ID=<organization-id>
export TF_VAR_infisical_client_id=<machine-identity-client-id>
export TF_VAR_infisical_client_secret=<machine-identity-client-secret>
export TF_VAR_infisical_project_id=<infisical-project-id>
export TF_VAR_infisical_host=https://app.infisical.com

terraform init -backend-config=backend.hcl
terraform workspace select -or-create staging
terraform plan -var-file=staging.tfvars
```

## Configuration reference

- `staging.tfvars` / `prod.tfvars`: non-sensitive knobs (scaling, image, port).
- GitHub Actions **secrets**: `SCW_ACCESS_KEY`, `SCW_SECRET_KEY`,
  `SCW_DEFAULT_PROJECT_ID`, `SCW_DEFAULT_ORGANIZATION_ID`,
  `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `INFISICAL_CLIENT_ID`,
  `INFISICAL_CLIENT_SECRET`.
- GitHub Actions **variables**: `TF_STATE_BUCKET`, `SCW_REGION`,
  `INFISICAL_PROJECT_ID`, `INFISICAL_HOST`.
- Infisical: application secrets, one environment per Terraform workspace.

No custom domain is configured: the app is reachable at the auto-generated
Scaleway URL (`terraform output container_url`). A domain can be added later
with a `scaleway_container_domain` resource.
